# 儲存庫規則

* 如果你看到任何違反這些規則的程式碼，請直接向我提出，而不是嘗試自行修正。
  * 在適用情況下，請建立 GitHub Issue。
* 嚴格遵守這些規則。

## 一般規則

* 如果能透過型別層級的紀律在執行期消除多餘的 try-catch 或 if-statement，就這麼做！
* 為你的型別、函式與類別命名得當。
  * 不要使用三個字母的縮寫。
  * 不要使用非標準縮寫。
  * 每個資料型別都有其意義，請選擇準確且具描述性的名稱。
  * 一般人應該只看函式簽章就能輕鬆理解函式在做什麼！
    * 有時會有例外。例如，你使用的是廣為理解的特定技術術語（saga、event 等）。
    * 通常你會覺得你的程式碼是規則的例外，但其實不是。

## 狀態、函式與類別

* 每個函式在相同輸入下都應產生相同輸出。也就是說，不要有隱藏狀態。
* 使用類別可防止固定狀態被任意（不安全地）變更；方法提供了安全地與狀態互動的方式。
* 如果你的邏輯不會變更固定狀態，那它很可能應該是獨立函式，而不是類別方法。
* 函式通常不應產生副作用（它們應該在計算上保持純粹）。
  * 例如，若你正在用 event 更新 state（計算上純粹），而你又想觸發 saga（計算副作用），請把觸發 saga 的邏輯放進 effect handler（可產生副作用的函式），再把它傳入原本計算純粹的函式，讓副作用可以安全觸發。

## Pydantic

* 閱讀 Pydantic 文件。
* 尊重 Pydantic 文件。
* pydantic 就足夠了。
* 針對你的使用情境宣告並重用中央 `ConfigDict`，通常你會希望 `frozen` 與 `strict` 為 `True`。

## 唯一 ID（UUID）產生

* 繼承 Pydantic 的 `UUID4` 類別來建立你自己的 UUID 類別。
* 在可行情況下，使用 `uuid.uuid4()` 以全新 UUID 初始化你的類別。
* 確保冪等性標籤是透過對持久化狀態做加鹽雜湊而產生。
  * 理由：如果節點崩潰並從較舊狀態恢復，不應在不同冪等性標籤下意外重複發佈同一事件。 
  * 每個不同函式都應使用唯一的鹽值，以避免冪等性標籤發生意外碰撞。

## 型別包裝

* 重用 Python 標準函式庫中已存在的型別。
* 當兩種不同資料型別在結構上相同（例如，不同 ID 都是 UUID 但不該混用）時，務必讓型別系統無法混淆它們。
  * 若你處理的是原始型別（`str`、`int` 等），請使用 `NewType`（執行期零額外成本）。
  * 若你處理的是可序列化資料物件，可考慮加入一個表示其型別的欄位（型別為 `str`）。

## 型別紀律

* 不要繞過型別檢查器，務必以任何必要手段維持嚴格型別。
* 預設情況下，在適合 enum 的地方使用字面值型別（例如 `Literal['one', 'two']`）。

專業提示：Python 的型別系統相當複雜且功能豐富，因此通常建議閱讀文件；Matt 在閱讀文件後發現，Python `typing` 函式庫允許你搭配 `Literal` 與 `get_args(type)` 檢查 `match` 是否完整實作。

## 使用 `@final`、凍結

* 在適用之處使用。

## 錯誤處理

* 不要無故 try-catch。
* 請確保你始終清楚你的程式碼所產生例外應在何處、何時被處理，避免變成令人措手不及的問題。
  * 一定要在 docstring 中寫下你錯誤處理的理由！
  * 在適當時機向同事說明細節。

## 相依性

* 未經詢問不要引入任何新相依套件。
* 不要要求那些在生產環境中不普遍的相依套件。
